# CLAUDE.md

本文件为 Claude Code 在 DeskTodoSeq 仓库中工作时提供指导。详细设计文档见 [详设说明.md](详设说明.md)，开发进度见 [项目进度.md](项目进度.md)。

## 项目概览

DeskTodoSeq — Windows 桌面任务管理工具，Python 3.10 + PySide6，Markdown 语法定义任务，SQLite + FTS5 存储，配备日历热力图。

## 常用命令

```bash
# 环境
uv venv --python 3.10 .venv && uv sync --dev

# 运行
uv run python main.py

# 测试
uv run pytest                                      # 全部 57 用例
uv run pytest -k "round_trip"                      # 关键字匹配

# 代码质量
uv run black src/ tests/ && uv run ruff check src/ tests/

# 打包：执行 build.bat（Nuitka standalone），安装版额外用 Inno Setup 编译 installer.iss
```

## 架构摘要

四层：`src/ui/` → `src/services/` → `src/models/` → SQLite，模块间通过 `SignalBus`（[src/utils/signal_bus.py](src/utils/signal_bus.py)）Qt 信号解耦通信。

核心原则：
- **raw_md 是规范数据源** — 结构化字段从 Markdown 解析派生，`MarkdownTaskFormatter.format()` 保证往返稳定
- **Design Tokens 统一配色** — `design_tokens.py` 的 `get_tokens()` 提供语义颜色，暗/亮主题同时适配
- **配置驱动** — `AppConfig`（JSON）集中管理设置，`config_changed` 信号通知热重载

PEP8：模块 `snake_case`，类 `PascalCase`，函数/变量 `snake_case`，常量 `UPPER_SNAKE_CASE`，私有 `_prefix`，Qt 信号过去式动词。

## 工作流程规则

**流程选择优先级**：BUG 修复或优化类问题优先按「规则 2 — 功能优化」流程处理；若在详设说明中未匹配到相关模块，则按「规则 1 — 新功能开发」流程处理。优化类问题定位后需明确说明是 BUG 还是扩展功能，并告知详设文档是否需要更新及更新方案。回答尽量使用中文。

### 0. 问题细分（前置步骤）

收到用户需求后，必须先进行问题细分，再按对应流程处理：

1. **分析定位**：仔细分析用户输入，定位涉及的功能模块和问题类型
2. **细分拆解**：将用户需求拆解为最小可处理单元，每个单元独立归类
3. **明确标注**：对每个子问题清晰标注是「新增功能」还是「优化类问题」（BUG / 扩展功能）
4. **分流处理**：新增功能按规则 1 处理，优化类问题按规则 2 处理；混合需求按子问题逐一跟进，不得混为一谈或跳过细分直接动手

### 1. 新功能开发

1. 先检索 [详设说明.md](详设说明.md)，确认是否已有相关模块
2. **无匹配**：按「需求 → 预览效果（含布局标注）→ 详细实现方案」编写设计文档，让用户确认效果后再开发
3. **有匹配**：标注为功能优化，走规则 2

### 2. 功能优化

优化类需求需先定位再动手，避免盲目改动引入回归：

1. **查阅定位**：先查阅 [详设说明.md](详设说明.md) 和 [项目进度.md](项目进度.md)，定位需求所属模块的详设、实现效果和历史变更记录
2. **问题分类**：定位后明确告知该问题是 **BUG**（代码行为与详设预期不符）还是 **扩展功能**（现有功能基础上新增行为），说明判断依据
3. **评估分析**：评估当前优化是否与已有功能冲突、是否需要重构、是否实为新增功能（若是则走规则 1）
4. **文档更新方案**：明确告知详设说明是否需要更新，以及更新方案是什么（新增章节 / 修改已有条目 / 补充说明等）
5. **推荐方案**：给出推荐优化方案，明确改动范围和影响，向用户确认（回答尽量使用中文）
6. **确认后执行**：
   - 先更新 [详设说明.md](详设说明.md)（设计文档先行）
   - 代码实现 → 自测（`timeout 5 uv run python main.py` 启动 + 功能验证）
   - 用户测试确认 → `git commit` 备份
   - 更新 [项目进度.md](项目进度.md)（记录优化时间、原因、影响范围）

### 3. 方案设计标准

设计方案或优化方案须达到以下标准：

1. **专业与前瞻**：准确识别模块间关联关系，评估改动对上下游的影响范围，方案需兼顾当前需求与未来扩展性
2. **业界主流**：优先选用业界广泛验证的成熟模式与组件，避免小众、复杂、维护成本高的实现方式，不重复造轮子
3. **方案确认**：方案需明确核心改动点、影响范围和潜在风险，向用户确认后再进入实现

### 4. 代码实现准则

借鉴业界成熟项目（Linux 内核小补丁、Google 小 CL、Django DEP 渐进交付）的工程实践：

1. **精准拆分**：将需求按依赖关系拆分为独立可验证的子任务，由易到难排序，每步完成后验证再推进
2. **小步提交**：每次改动聚焦单一逻辑变更，单次实现控制在可完成的时间窗口内，避免大范围修改导致定位困难或任务失败
3. **进度可见**：用 TodoWrite 跟踪子任务状态，实时反映进度，阻塞项及时暴露而非堆积到末尾
4. **快速验证**：每步完成后立即自测，不推迟到末尾，降低回归风险和排查成本

### 5. 通用准则

- **复用优先**：检索开源、可靠、可复用的组件，避免重复造轮子；推荐成熟设计方案，兼顾全局扩展性
- **主题适配**：UI 交互、配色严格遵循 `design_tokens.py` 的 `get_tokens()` 令牌体系，亮/暗双主题必须同时适配
- **全局一致**：Config 相关配置验证全局生效，需求不明主动向用户确认
- **环境隔离**：生产环境仅「功能演示分区」，打包以生产环境为准；开发/生产使用不同数据库和配置
- **文档同步**：[CLAUDE.md](CLAUDE.md)、[详设说明.md](详设说明.md)、[项目进度.md](项目进度.md) 任何变更后必须及时更新并 `git commit` 备份，确保三份文档始终反映项目最新状态
- **提交前文档检查**：每次功能更新或问题优化完成后，在 `git commit` 之前，必须检索代码实现内容，与 [详设说明.md](详设说明.md) 逐项对比，查缺补漏，确保文档与代码逻辑始终一致；同时更新 [项目进度.md](项目进度.md) 记录变更

### 6. 测试与备份

1. 自测通过后告知用户"自测完成"，由用户测试确认
2. 用户确认问题已解决后，**必须先 `git commit` 提交代码作为备份**，再进行下一个任务

### 7. 验证确认与提交

1. **主动追问**：问题实现并自测通过后，告知用户验证。若用户未明确反馈或反馈模糊，必须主动追问，不得默认通过
2. **确认标准**：必须等到用户清晰表述问题"已解决"或"验证通过"后，方可执行 git commit
3. **批量任务提交**：同一模块的一批任务可合并为一次提交，但提交前必须确认全部任务均已通过用户验证，避免遗漏或未验证即提交
